# M2-07 ‚Äî Brownfield onboarding

**Milestone:** 2 ‚Äî Real product
**Estimated effort:** 1 day
**Dependencies:** M1-01 (CLI), M1.1-01 (init config templates), M2-06 (dbt integration), M2-08 (convention inference)

## Purpose

Most users of Carve will have an existing dbt project. The `carve init` flow needs to detect this, integrate without overwriting, and produce a useful state on first run. Greenfield init is the easy case (already shipped in M1-01 + M1.1-01); brownfield is the one that determines whether Carve is adoptable.

## The greenfield case (already shipped)

For comparison: `carve init` in an empty directory produces this layout today (M1-01 + M1.1-01):

```
.
‚îú‚îÄ‚îÄ carve.toml
‚îú‚îÄ‚îÄ carve/
‚îÇ   ‚îú‚îÄ‚îÄ connections.toml         (commented template, M1.1-01)
‚îÇ   ‚îú‚îÄ‚îÄ models.toml              (commented template, M1.1-01)
‚îÇ   ‚îú‚îÄ‚îÄ runner.toml              (commented template, M1.1-01)
‚îÇ   ‚îî‚îÄ‚îÄ agents/                  (empty dir, populated by user / later specs)
‚îú‚îÄ‚îÄ pipelines/                   (empty dir; pipeline runs land here)
‚îú‚îÄ‚îÄ .carve/
‚îÇ   ‚îî‚îÄ‚îÄ state.db                 (initialized SQLite state store)
‚îú‚îÄ‚îÄ .env.example                 (commented template, M1.1-01)
‚îî‚îÄ‚îÄ .gitignore                   (with .env, .carve/, *.sqlite entries)
```

Notes on what the greenfield layout deliberately does **not** include today:

- No `carve/server.toml` ‚Äî the config loader will read one if a user adds it, but no spec scaffolds it. M2-10 (FastAPI server) is the natural place to introduce it.
- No `carve/conventions.md` ‚Äî produced by M2-08 convention inference, which runs as part of brownfield init (this spec) or via an explicit `carve infer-conventions` call. Greenfield projects start with no conventions document; the agent treats absence as "no project conventions yet".
- No `dbt/` scaffold ‚Äî Carve does not generate a dbt project for the user. Greenfield Carve users who also need a new dbt project should run `dbt init` themselves; the brownfield flow below is what wires an existing dbt project into Carve.
- No `snowflake/` directory.

The user's natural setup flow is still the M1.1-03 path: `carve init` ‚Üí `cp .env.example .env` ‚Üí fill in values ‚Üí `carve plan ...`.

## The brownfield case

`carve init` in a directory that already has `dbt_project.yml` (or in any parent of it):

1. **Detect** existing dbt project
2. **Locate** the `dbt_project.yml`, `profiles.yml`, manifest
3. **Read** the existing config without modifying it
4. **Generate** Carve's own config files (the same set M1.1-01 already writes), but with the `[dbt]` and `[detection]` sections in `carve.toml` populated, and `connections.toml` pre-filled from `profiles.yml`
5. **Run `dbt parse`** to ensure manifest is current
6. **Run convention inference** (M2-08) to generate `carve/conventions.md`
7. **Append** Carve-specific entries to `.gitignore` (or create one if missing)
8. **Don't touch** anything else

Files added by this spec on top of the M1.1-01 greenfield layout, in brownfield mode:

- `carve/conventions.md` (via M2-08 inference run)
- `[dbt]` and `[detection]` sections in `carve.toml`
- A pre-filled (rather than fully commented) `carve/connections.toml`, with sensitive values rewritten as `${ENV_VAR}` placeholders
- `.env.example` gains entries for any env vars referenced by the auto-filled connections

## Detection logic

In `src/carve/cli/commands/init/detection.py`:

```python
def detect_dbt_project(start_dir: Path) -> DbtProjectInfo | None:
    """Walk up from start_dir looking for dbt_project.yml."""
    current = start_dir.resolve()
    for _ in range(10):  # max 10 levels up
        candidate = current / "dbt_project.yml"
        if candidate.exists():
            return DbtProjectInfo(
                project_dir=current,
                project_yml=candidate,
                profiles_yml=find_profiles_yml(current),
                manifest=find_manifest(current),
            )
        if current.parent == current:
            break
        current = current.parent
    return None

def find_profiles_yml(project_dir: Path) -> Path | None:
    # Order: project_dir/profiles.yml, then ~/.dbt/profiles.yml
    candidates = [
        project_dir / "profiles.yml",
        Path.home() / ".dbt" / "profiles.yml",
    ]
    return next((p for p in candidates if p.exists()), None)
```

## Three brownfield scenarios

### Scenario 1: Carve in same repo as dbt

Most common. User runs `carve init` in their dbt project's repo root (or in a subdirectory, in which case Carve walks up).

```
my-dbt-repo/
‚îú‚îÄ‚îÄ dbt_project.yml          ‚Üê existing
‚îú‚îÄ‚îÄ models/
‚îú‚îÄ‚îÄ carve.toml               ‚Üê new (with [dbt] section)
‚îî‚îÄ‚îÄ carve/                   ‚Üê new (same shape as M1.1-01 greenfield, plus conventions.md)
    ‚îú‚îÄ‚îÄ connections.toml     (auto-filled from profiles.yml)
    ‚îú‚îÄ‚îÄ models.toml
    ‚îú‚îÄ‚îÄ runner.toml
    ‚îú‚îÄ‚îÄ agents/
    ‚îî‚îÄ‚îÄ conventions.md       (from M2-08 inference)
```

Carve config references `./` as the dbt project root.

### Scenario 2: Sibling directory

User wants Carve in a separate directory next to the dbt project:

```
~/work/
‚îú‚îÄ‚îÄ my-dbt/
‚îÇ   ‚îî‚îÄ‚îÄ dbt_project.yml
‚îî‚îÄ‚îÄ my-carve/                ‚Üê user runs carve init here
    ‚îî‚îÄ‚îÄ carve.toml
```

The init flow asks: "I detected a dbt project at `../my-dbt`. Use it?" If yes, Carve config has:

```toml
[dbt]
project_dir = "../my-dbt"
```

### Scenario 3: Remote git URL

User wants Carve to manage a dbt project they don't have a local clone of. (Less common, mostly relevant for SaaS later.) For OSS v0.1, defer this ‚Äî the user must clone first.

## Init flow with brownfield detection

The current `init.command` (M1.1-01) writes the greenfield layout unconditionally. This spec wraps it with detection:

```python
def init_command(
    interactive: bool = True,
    dbt_project_path: str | None = None,
):
    # 1. Detect
    detected = detect_dbt_project(Path.cwd())

    if dbt_project_path:
        # User passed --dbt-path
        detected = DbtProjectInfo.from_path(dbt_project_path)
    elif detected and interactive:
        confirm = typer.confirm(
            f"Detected dbt project at {detected.project_dir}. Use it?",
            default=True
        )
        if not confirm:
            detected = None

    # 2. Generate Carve config
    if detected:
        write_brownfield_config(detected)   # extends the M1.1-01 templates
    else:
        write_greenfield_config()           # the existing M1.1-01 path

    # 3. Initialize state store (already part of the M1.1-01 path)
    initialize_database(...)

    # 4. If brownfield, parse manifest and infer conventions
    if detected:
        run_dbt_parse(detected)
        infer_conventions(detected)         # M2-08; produces carve/conventions.md

    # 5. Update .gitignore (greenfield already writes one; brownfield appends)
    update_gitignore()

    # 6. Print next steps
    print_post_init_message(detected)
```

The greenfield branch should call into the existing M1.1-01 implementation (factor the current `init.command` body into a `write_greenfield_config()` so this spec doesn't duplicate it).

## Generated config for brownfield

`carve.toml` ‚Äî the M1.1-01 base, plus `[dbt]` and `[detection]` sections:

```toml
# carve.toml
[project]
name = "<directory_name>"
version = "0.0.1"
default_target = "dev"

[paths]
config_dir = "carve"

[dbt]
project_dir = "<detected_path>"
profiles_dir = "<detected_profiles_path>"
target = "dev"  # set to the user's chosen default during the profile-selection prompt

[detection]
mode = "brownfield"
detected_at = "<timestamp>"
existing_models = <count>
existing_sources = <count>
```

The `[detection]` section is metadata for the user's reference ‚Äî what was detected at init time.

`carve/connections.toml`, `carve/models.toml`, `carve/runner.toml` ‚Äî the M1.1-01 commented templates are still the baseline. Brownfield mode replaces `connections.toml` with an auto-filled version (see "Existing connections detection" below); `models.toml` and `runner.toml` remain the M1.1-01 commented templates.

## What carve init never modifies

A list of files and patterns Carve will not touch in brownfield mode:

- `dbt_project.yml`
- `profiles.yml`
- `packages.yml`, `dependencies.yml`
- Any file in `models/`, `tests/`, `seeds/`, `macros/`, `snapshots/`, `analyses/`
- `.dbt/`, `target/`, `dbt_packages/`
- The user's existing `.gitignore` (only appends; never overwrites)
- `requirements.txt`, `pyproject.toml`, or any other dependency manifest

If Carve would create a file that already exists, it skips and warns (the existing `_write_if_missing` helper from M1-01 already handles this). No clobbering, ever.

## Existing connections detection

If `profiles.yml` exists, Carve reads it (read-only) to suggest connection config. Instead of writing the M1.1-01 fully-commented `connections.toml`, it writes an auto-filled version.

### Profile selection

`profiles.yml` typically defines one *profile* (named after the dbt project) with one or more *outputs* (`dev`, `prod`, `qa`, etc.). Carve handles three cases:

1. **One profile, one output.** No prompt; auto-fill that single output as `[snowflake.<output_name>]` and set `default_target = "<output_name>"` in `carve.toml`.

2. **One profile, multiple outputs.** Prompt the user with the list of outputs and a multi-select:
   ```
   Detected profile "my_project" with multiple outputs:
     [x] dev   (account=xy12345.us-east-2, role=TRANSFORMER_DEV)
     [ ] prod  (account=xy12345.us-east-2, role=TRANSFORMER_PROD)
     [ ] qa    (account=xy12345.us-east-2, role=TRANSFORMER_QA)
   Which targets should Carve manage? (space to toggle, enter to confirm)
   ```
   Defaults to whatever output `target:` points at in the profile (typically `dev`). Each selected output becomes a `[snowflake.<output>]` block. `default_target` is set to whichever the user marks as default (or `target:` from the profile if only one is selected).

3. **Multiple profiles.** Prompt the user to pick one profile first, then fall through to (1) or (2):
   ```
   Detected multiple profiles in profiles.yml:
     > my_project   (3 outputs)
       legacy_etl  (1 output)
       experiments (2 outputs)
   Which profile should Carve use?
   ```

The prompt uses Carve's existing CLI input pattern (rich-based, single-keystroke navigation). For non-interactive environments (`CARVE_INIT_NONINTERACTIVE=1` or `--yes`), Carve picks the profile that matches the `dbt_project.yml`'s `profile:` field, then picks the output that matches the profile's `target:` field ‚Äî both deterministic, both the dbt-default behavior. If those resolutions fail, init exits with a clear error asking the user to re-run interactively or pass explicit flags (`--profile <name> --outputs <list>`).

### Sample profile layout

```yaml
# Detected profiles.yml:
my_project:
  target: dev
  outputs:
    dev:
      type: snowflake
      account: xy12345.us-east-2
      user: dev_user
      role: TRANSFORMER_DEV
      ...
```

Becomes a suggestion in `carve/connections.toml`:

```toml
# Auto-generated from profiles.yml ‚Äî review and adjust as needed.
# Every value is sourced from .env so credentials and account-shape data
# stay out of source control. See .env.example for the env var list.
[snowflake.dev]
account   = "${SNOWFLAKE_ACCOUNT}"
user      = "${SNOWFLAKE_USER}"
password  = "${SNOWFLAKE_PASSWORD}"
role      = "${SNOWFLAKE_ROLE}"
warehouse = "${SNOWFLAKE_WAREHOUSE}"
database  = "${SNOWFLAKE_DATABASE}"
```

**Every** Snowflake connection field is rewritten as a `${ENV_VAR}` placeholder ‚Äî including `account`, `role`, `warehouse`, and `database`, not just credentials. Reasoning: the boundary between "sensitive" and "non-sensitive" is fuzzy at adoption time (account names leak environment topology; role names hint at internal RBAC structure), and a single rule ‚Äî "everything goes to .env" ‚Äî is easier to teach and audit than per-field judgement calls. The auto-generated `.env.example` extends the M1.1-01 baseline with one entry per placeholder Carve emitted (per target ‚Äî so `SNOWFLAKE_DEV_ACCOUNT`, `SNOWFLAKE_PROD_ACCOUNT`, etc., when multiple targets are detected and the user chooses to pull more than one).

For multi-target profiles where the user picks a single profile during init (see "Existing connections detection" above), the env vars use the unprefixed `SNOWFLAKE_*` form. If the user opts to pull multiple targets in one run, Carve emits target-prefixed env vars to avoid collisions.

## .gitignore additions

The M1.1-01 greenfield path already writes a `.gitignore` containing `.env`, `.carve/`, `*.sqlite`, `*.sqlite3`. Brownfield mode must respect a pre-existing `.gitignore` and append a clearly-marked block instead:

```
# === Carve ===
.carve/
.env
```

If `.gitignore` doesn't exist, brownfield falls back to the greenfield-style `.gitignore` write.

## Tests

- Detection finds dbt_project.yml in current dir
- Detection finds dbt_project.yml in parent dir
- `--dbt-path` overrides detection
- Decline of interactive confirmation falls through to greenfield (the existing M1.1-01 path)
- Brownfield mode never overwrites existing files
- profiles.yml is read for connection suggestions
- Sensitive values become env var placeholders
- `.gitignore` is appended to, not replaced (when one already exists)
- Brownfield init in a directory with a pre-existing `.gitignore` containing unrelated entries leaves those entries intact and adds the `# === Carve ===` block

Use temporary directories with various dbt project layouts as fixtures.

## Acceptance criteria

- `carve init` against an existing dbt project produces working Carve config without modifying the dbt project
- Detection handles same-repo and sibling-dir cases
- Convention inference (M2-08) runs as part of brownfield init and produces `carve/conventions.md`
- Greenfield init continues to behave exactly as it did post-M1.1-01 (no regressions to the existing layout, templates, or state-store init)
- Post-init message is actionable
