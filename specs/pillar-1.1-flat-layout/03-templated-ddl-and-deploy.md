# P1.1-03 — Templated DDL + simplified deploy

**Milestone:** Pillar 1.1 — Flat layout + git-based promotion
**Estimated effort:** 1 day
**Dependencies:** P1.1-01 (flat layout), P1.1-02 (destination.toml with sections)
**Lineage:** Supersedes **P1-06** (per-EL DDL snapshot — concrete identifiers per target) and **P1-08** (`carve el deploy --from X --to Y` with file copy + DDL apply). Replaces the per-target SQL snapshot with a single Jinja template rendered at deploy time, and reframes `deploy` as a single-target DDL-readiness verb.

## Purpose

Two interlocking changes that together make Carve's deploy match how data shops actually promote code in 2026:

1. **DDL becomes a Jinja template.** One `el/<name>/snowflake.sql.j2` lives in git, target-agnostic. At deploy time, Carve renders it with the target's resolved database/schema/table/runtime-role identifiers. The rendered SQL is what runs against Snowflake. No more "ANALYTICS_DEV" / "ANALYTICS_PROD" snapshots in source.

2. **`carve el deploy` operates on one target.** `--from X --to Y` is gone. The new verb is `carve el deploy <name> --target X` and it does **only** the DDL-readiness work: pre-flight against X, render DDL for X, apply via X's deploy role, smoke-verify with X's runtime role. No file copy — git is the promotion mechanism, not Carve.

## CLI surface

```
carve el deploy <name> --target X
  [--yes / -y]                     # skip the confirmation prompt
  [--no-smoke-test]                # skip post-DDL verification
  [--no-auto-fix]                  # skip recovery agent
  [--max-fix-attempts N]           # per-failure-event budget
  [--dry-run]                      # render + validate + show; do not apply
```

`carve el verify <name> --target X` stays largely as it is in P1-08: connects with X's runtime role and asserts the destination state is what the artifact expects. Now reads the rendered DDL identifiers from `destination.toml`'s resolution-for-target output (per P1.1-02).

## Template format

```jinja
{# el/iowa/snowflake.sql.j2

   Auto-generated by `carve build` and re-rendered by `carve el deploy
   <name> --target X` at deploy time.

   Available variables:
     {{ database }}       Resolved per P1.1-02's destination.toml + env-var rules.
     {{ schema }}         Same.
     {{ table }}          Always the literal from destination.toml.
     {{ runtime_role }}   The runtime role from [snowflake.X] in connections.toml.
     {{ target }}         The target name, lowercase (e.g. "prod").

   Every statement is idempotent — same allow-list as P1-06 (P1-08's
   `validate_ddl_statements`).
#}

-- === Schema ===
CREATE SCHEMA IF NOT EXISTS {{ database }}.{{ schema }};

-- === Table ===
CREATE TABLE IF NOT EXISTS {{ database }}.{{ schema }}.{{ table }} (
    INVOICE_LINE_NO  VARCHAR(50)   NOT NULL,
    DATE             DATE,
    STORE_NUMBER     VARCHAR(20),
    ITEM_DESCRIPTION VARCHAR(255),
    BOTTLES_SOLD     NUMBER(10,0),
    SALE_DOLLARS     NUMBER(12,2),
    LOCATION         VARIANT,
    _LOADED_AT       TIMESTAMP_NTZ DEFAULT CURRENT_TIMESTAMP(),
    PRIMARY KEY (INVOICE_LINE_NO)
);

-- === Grants ===
GRANT SELECT, INSERT, UPDATE, DELETE
    ON TABLE {{ database }}.{{ schema }}.{{ table }}
    TO ROLE {{ runtime_role }};
```

### Template variables

| Variable | Source | Notes |
|---|---|---|
| `database` | P1.1-02 resolution (section → default → env var) | Snowflake unquoted identifier; validated before render. |
| `schema` | P1.1-02 resolution | Same. |
| `table` | P1.1-02 resolution (always required) | Same. |
| `runtime_role` | `config.connections.snowflake[<target>].role` | The role that runs `carve el run`. The deploy role doesn't appear in DDL — it's the role *applying* the DDL, not the role being granted to. |
| `target` | The active target name, lowercased | For docstrings / comments inside the rendered SQL. Not typically interpolated into identifiers. |

No other variables. The template engine is **sandboxed** Jinja2 (`SandboxedEnvironment`) with `autoescape=False` (SQL has no HTML semantics) and **no access to environment-introspection filters**. The render context is the four variables above plus standard Jinja built-ins — nothing more.

### Identifier validation at render time

Before substitution, the renderer validates `database` / `schema` / `table` / `runtime_role` against P1-08's `validate_identifier` (Snowflake unquoted-identifier grammar: `^[A-Za-z_][A-Za-z0-9_]*$`). Invalid identifiers raise `InvalidSnowflakeIdentifierError` *before* the template renders — closes the same SQL-injection vector P1-08 closed for the per-target snapshot.

### Post-render allow-list

The rendered SQL passes through P1-08's `parse_ddl_statements` + `validate_ddl_statements` (the DDL allow-list) before any statement executes. Same rules as today: `CREATE ... IF NOT EXISTS`, `GRANT`, `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` are allowed; `CREATE OR REPLACE`, bare `DROP`, bare `RENAME`, DML inside DDL, `DROP DATABASE` are rejected. A malformed template that emits a forbidden statement type fails the deploy with `UnsafeDdlError`.

## Deploy command flow (new shape)

`carve el deploy <name> --target X`:

1. **Resolve the active target.** Same as P1-07 / P1-08 — `--target X` wins; falls back to top-level `--target` flag → `CARVE_TARGET` → `default_target`.
2. **Validate inputs.** Artifact dir at `el/<name>/` exists; target X has a `[snowflake.X]` section in `connections.toml`; deploy role at `[snowflake.X_deploy]` exists (or the user opted out — TBD: see "Open: deploy role required?" below).
3. **Resolve destination identifiers for X.** Read `el/<name>/destination.toml`, apply P1.1-02's resolution-for-target rule, get `database` / `schema` / `table`. Read `runtime_role` from `[snowflake.X].role`. Validate all four against the identifier grammar.
4. **Render the template.** Load `el/<name>/snowflake.sql.j2`; render with the four variables. If the template doesn't exist (artifact predates P1.1-03), fall back to `el/<name>/snowflake.sql` as a verbatim file — deprecation warning instructing the user to convert. (One-version transition; removed in v0.2.)
5. **Pre-flight against X.** Connect with X's deploy role; run the same drift checks P1-08 runs (schema exists, table exists with expected columns, runtime role has grants, runtime role itself exists). Surface drift; recovery agent may attempt fixes within budget (per P1-09).
6. **Confirmation prompt** (`--yes` skips). Shows:
   - The destination FQN: `<database>.<schema>.<table>`
   - The runtime role being granted to
   - Drift (if any) detected at pre-flight
   - The rendered DDL — abbreviated to statement headlines (`CREATE SCHEMA IF NOT EXISTS ...`, `CREATE TABLE IF NOT EXISTS ...` plus column count, `GRANT ... TO ROLE ...`) with a "Show full SQL" toggle (`carve el deploy <name> --target X --dry-run` is the non-interactive way to see it all).
   - destination.toml override warnings from P1-08 stage 2 (preserved unchanged).
7. **Apply DDL.** Statements run sequentially under the deploy role. Each is validated against the allow-list right before it executes. On any failure the recovery agent (P1-09) may attempt a fix within the budget.
8. **Smoke verify.** Connect with X's runtime role; assert the table exists with the right columns; the runtime role can `SELECT` against the table; grants present.
9. **Record a Run row** of `kind="deploy"`, `target=X`. Manifest_json drops the file list — there are no copied files now — but keeps the rendered DDL statement count and the rendered identifier triple for audit.
10. **Print summary**: rendered destination FQN, statements applied vs no-op, verify result, recovery summary if any phase invoked the agent.

`--dry-run` runs steps 1–6 and prints the full rendered SQL; never connects to apply, never invokes recovery, never creates a Run row. Useful for review-the-rendered-DDL-in-CI workflows.

## What goes away

- **File-copy phase.** `copy_artifact` and `copy_ddl_file` in `src/carve/core/deploy/copier.py` are deleted. Symlink-rejection logic isn't lost — it's not relevant when no copy is happening.
- **`--from X` flag.** The deploy command takes one target, not two.
- **Build's `manifest_json["files"]` listing artifact files.** The build still writes files to `el/<name>/`; the manifest now just records what *was* written for that build (used by recovery's "the build wrote these files" context). No deploy uses it.
- **Per-target snowflake/ directory.** `targets/<active>/snowflake/<artifact>.sql` is gone. The DDL lives at `el/<name>/snowflake.sql.j2`.

## Build flow updates

`carve build <plan_id>`:

- Continues to emit `el/<name>/main.py` and `el/<name>/requirements.txt` (per P1.1-01).
- Now emits `el/<name>/snowflake.sql.j2` instead of a per-target `.sql` file. The build agent generates the template directly — the system prompt says "write SQL with `{{ database }}` / `{{ schema }}` / `{{ table }}` / `{{ runtime_role }}` placeholders; never inline concrete identifiers." The connection-context preamble's DDL section (which used to show resolved values) becomes the template-variables guide.
- destination.toml emission (per P1.1-02) is unchanged.
- The build's `target` field on the Build row still records "which target's catalog informed the build" — the agent inspects the active target's existing schema/tables via P1-05's catalog skills, which biases the design. This is *not* "the target this build will deploy to"; deploys are independent of which target the build was authored against.

## Deploy role lookup

P1-08 used `pool.get(f"{target}_deploy")` to find the deploy role's connection. P1.1-03 keeps that convention. Open question: should it be required?

**Open: deploy role required?**

Two options:
- **Strict**: every target needs an `[snowflake.<target>_deploy]` section in `connections.toml`. If absent, `carve el deploy` exits 2 with a clear error: "Add `[snowflake.<target>_deploy]` to `connections.toml` (see `docs/deploy-roles.md`)."
- **Lenient**: if `[snowflake.<target>_deploy]` is absent, fall back to `[snowflake.<target>]` (runtime role with DDL privileges). Warn the user that this conflates privileges — fine for solo dev / sandbox accounts, dangerous in shared environments.

I'd lean **strict** with a one-shot upgrade verb (`carve target create <name>_deploy --copy-from <name>`) to scaffold the deploy section. Solo-dev / sandbox users hit the error once, run the scaffold command, and continue. Shared-environment users *want* the explicit error.

Defer this to the implementation phase; if it's controversial, add a `--allow-runtime-role-for-deploy` flag as a safety valve.

## Tests

- `test_template_renders_with_validated_identifiers` — `el/iowa/snowflake.sql.j2` with the four placeholders renders to a concrete SQL string against a target config.
- `test_template_rejects_invalid_database_identifier` — `database = "bad-name"` → `InvalidSnowflakeIdentifierError` BEFORE render.
- `test_template_uses_sandboxed_environment` — a template that tries to access `__class__` or `__globals__` raises a Jinja security error.
- `test_rendered_ddl_passes_allow_list` — typical generated template (CREATE SCHEMA IF NOT EXISTS / CREATE TABLE IF NOT EXISTS / GRANT) passes `validate_ddl_statements`.
- `test_rendered_ddl_rejects_create_or_replace_in_template` — a template containing `CREATE OR REPLACE TABLE` fails `validate_ddl_statements` AFTER render (defense-in-depth: the prompt already forbids this).
- `test_deploy_runs_against_single_target` — `carve el deploy iowa --target prod --yes` connects with prod's deploy role, applies rendered DDL, smoke-verifies; no `--from` accepted.
- `test_deploy_drops_from_flag` — `carve el deploy iowa --from dev --to prod` exits 2 with typer's "no such option" for `--from`. Deprecation pointer in the error if feasible.
- `test_deploy_dry_run_renders_and_validates_no_connect` — `--dry-run` prints the rendered SQL, exits 0, doesn't open a deploy-role connection.
- `test_deploy_dry_run_surfaces_invalid_identifiers` — `--dry-run` catches identifier validation issues before any connect attempt.
- `test_deploy_missing_template_falls_back_to_legacy_sql_file` — when `snowflake.sql.j2` is absent but `snowflake.sql` exists, deploy uses the verbatim file with a deprecation warning. (Removed in v0.2.)
- `test_build_writes_sql_j2_template` — `carve build <plan_id>` writes `el/<name>/snowflake.sql.j2` (not a per-target `.sql`).
- `test_build_agent_emits_template_not_concrete_identifiers` — the build agent's emitted template contains `{{ database }}` not `ANALYTICS_DEV`.
- `test_verify_uses_rendered_destination_for_target` — `carve el verify iowa --target staging` renders DDL identifiers for staging, asserts state against staging's runtime role.
- `test_recovery_agent_edits_template_not_per_target_sql` — recovery context #3 (DDL apply failure) edits `el/<name>/snowflake.sql.j2` (P1.1-04 wires the path; this spec just changes what's at that path).

## Acceptance criteria

- `carve el deploy <name> --target X` works against a single target; `--from` is rejected.
- DDL lives in `el/<name>/snowflake.sql.j2` and is rendered per-target at deploy time.
- The rendered SQL passes P1-08's allow-list before any statement executes; the allow-list is unchanged.
- Identifier validation rejects bad `database` / `schema` / `table` / `runtime_role` values BEFORE render.
- `--dry-run` shows the full rendered SQL without connecting to apply.
- The build flow emits a template, not a concrete-identifier snapshot.
- Recovery agent context #3 (DDL apply) edits the template; the next attempt re-renders.
- Legacy `snowflake.sql` (verbatim, pre-P1.1-03) works with a deprecation warning for one minor version.
- All P1-06 + P1-08 tests retargeted to the new flow pass.
- `ruff` + `mypy --strict` + `pytest` stay green.

## Out of scope

- A `--from X --to Y` "snapshot promotion" verb. Git handles this; we don't.
- Multi-target deploys in a single invocation (`--targets dev,staging,prod`). Defer.
- Approving destructive DDL inside a template (DROP COLUMN, etc.). Stays out of scope per P1-06's contract; the template can express destructive intent the same way the snapshot did, and the same plan-time-approval rule applies.
- Detecting template renders that would emit identical SQL across two targets and warning "this is target-agnostic; consider hardcoding." Premature.
- A `carve el deploy diff <name> --target X` verb showing the SQL diff vs what's actually applied in X. Useful but defer.
- Schema migration files (versioned DDL changes that can't be expressed as idempotent statements). Defer to Pillar 2.

## What this enables

- The `el/<name>/` tree is now genuinely target-agnostic. Every file in it (script, requirements, destination, DDL template) runs against any target by swapping env vars.
- CI/CD is straightforward: a post-merge job runs `carve el deploy <name> --target prod --yes` from the merged checkout, using prod's deploy creds (not dev's). A scheduled job runs `carve el run <name> --target prod` from the same checkout with prod's runtime creds.
- The deploy command becomes inspectable: `--dry-run` shows the full rendered SQL before any state is touched. PR reviewers see what's about to run against prod.
- The privilege envelope is unambiguous: deploy needs deploy creds, run needs runtime creds, and they never share a machine in CI.
- Future pillars (Pillar 2 dbt models, Pillar 3 pipeline steps) inherit the same shape: target-agnostic source tree, target-specific identifiers rendered at apply time.
