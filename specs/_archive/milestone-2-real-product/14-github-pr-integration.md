# M2-14 — GitHub PR integration

**Milestone:** 2 — Real product
**Estimated effort:** 1 day
**Dependencies:** M2-01 (plan/deploy), M2-10 (server)

## Purpose

When a plan is deployed, generated code lands as a pull request against the user's repo, not as a direct commit to main. The PR description includes the plan summary, file diffs, and impact analysis. CI runs on the PR; the user reviews; the user merges. This is the safety review boundary.

## Why GitHub-only for v0.1

- GitHub is dominant in the data ecosystem; covers ~80% of users
- GitLab support adds complexity (different API, OAuth flow); save for v0.2
- Bitbucket usage in data tooling is small; defer
- Self-hosted Git (no PRs) goes to a "patches branch" mode (just a branch, no PR); covered later

## Authentication

Two paths:

### Path 1 — GitHub Personal Access Token (PAT)

User creates a PAT with `repo` scope, sets it in `.env`:

```
GITHUB_TOKEN=ghp_xxx
```

Carve config references it:

```toml
[git]
provider = "github"
token = "${GITHUB_TOKEN}"
default_branch = "main"
```

This is the v0.1 path. Simple, no OAuth flow.

### Path 2 — GitHub App (deferred)

For SaaS, a GitHub App lets users install Carve on specific repos with proper scope. Token rotation, audit logs. Out of scope for v0.1.

## Configuration

`carve/git.toml`:

```toml
[git]
provider = "github"
token = "${GITHUB_TOKEN}"
remote = "origin"  # which git remote to push to
default_branch = "main"  # PR target

[git.pr]
draft = false  # open PRs as drafts
title_prefix = "[carve]"
labels = ["carve-generated", "auto-pr"]
assign_to = []  # GitHub usernames to auto-assign
request_reviewers = []  # GitHub usernames or teams

[git.branch]
prefix = "carve/"  # all branch names start with this
naming = "carve/<plan_id>-<slug>"  # template
```

## The PR creation flow

After `carve deploy` completes successfully:

1. Determine remote and target branch from config
2. Determine new branch name from template
3. Check the working tree is clean (no uncommitted changes outside the deploy)
4. Create the new branch from current main: `git checkout -b carve/<plan_id>-<slug>`
5. Stage all generated/modified files
6. Commit with a structured message (see below)
7. Push the branch
8. Open a PR via the GitHub API

If any step fails, the deploy is still considered successful (the code lives locally), but the PR step is marked as failed and the user gets a clear instruction to push manually.

## Branch naming

`carve/<plan_id>-<short-slug>` where slug is derived from the goal.

Examples:
- `carve/plan_a3f29-make-stg-orders-incremental`
- `carve/plan_b7c12-onboard-salesforce`

Slug generation: lowercase, dashes, max 50 chars after the plan ID.

## Commit message

Structured for machine and human readability:

```
[carve] Make stg_orders incremental

Plan: plan_a3f29
Goal: make stg_orders incremental with order_id as unique key

Changes:
- Modified: dbt/models/staging/stg_orders.sql
- Modified: dbt/models/staging/_staging.yml
- Created: dbt/tests/test_stg_orders_incremental.sql

Generated by Carve v0.0.5
Deploy run: run_xx88aa
```

The Plan: line is what allows the UI to link back to the plan from a commit later.

## Implementation

`src/carve/core/git/github.py`:

```python
class GitHubProvider:
    def __init__(self, config: GitConfig):
        self.config = config
        self.gh = Github(auth=Auth.Token(config.token))

    def open_pr(
        self,
        plan: Plan,
        run: Run,
        diff: list[FileDiff],
        repo_root: Path,
    ) -> str:
        # 1. Detect remote
        remote_url = self._git_remote_url(repo_root)
        owner, repo_name = self._parse_github_url(remote_url)

        # 2. Create branch
        branch = self._branch_name(plan)
        self._create_branch(repo_root, branch)

        # 3. Commit
        message = self._commit_message(plan, run, diff)
        self._commit_all(repo_root, message)

        # 4. Push
        self._push(repo_root, branch)

        # 5. Open PR
        title = self._pr_title(plan)
        body = self._pr_body(plan, run, diff)
        repo = self.gh.get_repo(f"{owner}/{repo_name}")
        pr = repo.create_pull(
            title=title,
            body=body,
            head=branch,
            base=self.config.default_branch,
            draft=self.config.pr.draft,
        )

        # 6. Apply labels, reviewers
        if self.config.pr.labels:
            pr.add_to_labels(*self.config.pr.labels)
        if self.config.pr.request_reviewers:
            pr.create_review_request(reviewers=self.config.pr.request_reviewers)

        return pr.html_url
```

## Local git operations

Use `subprocess` for git commands rather than `pygit2` or `dulwich`. Reasoning:

- The user's git is configured the way they like (signing keys, hooks, etc.)
- Bypassing it would require recreating all that
- Performance is not a concern for the volume of operations Carve needs

```python
def _create_branch(self, repo_root: Path, branch: str):
    subprocess.check_call(["git", "checkout", "-b", branch], cwd=repo_root)

def _commit_all(self, repo_root: Path, message: str):
    subprocess.check_call(["git", "add", "-A"], cwd=repo_root)
    subprocess.check_call(["git", "commit", "-m", message], cwd=repo_root)

def _push(self, repo_root: Path, branch: str):
    remote = self.config.remote
    subprocess.check_call(["git", "push", "-u", remote, branch], cwd=repo_root)
```

## Error handling

Common failures and recovery:

- **Working tree dirty before deploy**: refuse. Tell user to commit or stash first.
- **Branch already exists**: append a numeric suffix. `carve/plan_a3f29-...`, then `-2`, etc.
- **No remote configured**: skip PR step, save changes to a local branch, instruct user to push manually.
- **Push fails (auth, network)**: keep the branch locally, report the error, save the deploy as success but PR as failed.
- **GitHub API rate limit**: wait + retry with backoff up to 60s.
- **PR API failure (e.g., 422 from a missing label)**: open the PR without the label and warn.

## "no PR" mode

For users without a remote (just a local repo), a `--no-pr` flag on deploy (or `[git.pr]` `enabled = false`) skips PR creation. Branch + commit happens; that's it.

For users on self-hosted GitLab or Bitbucket, this is the v0.1 fallback. Manual PR creation, but at least the branch is ready.

## Tests

- Branch naming generates expected slugs
- Commit message includes plan ID
- PR body renders correctly given plan + diff
- GitHub API errors surface clearly
- Dirty working tree blocks deploy with helpful message
- `--no-pr` mode commits without API calls

Use `pygithub`'s test fixtures or VCR-style cassette recordings for API tests.

## Acceptance criteria

- After `carve deploy`, a PR is opened against the configured default branch
- PR description includes plan summary, diff, impact, link back to run
- Failures in the PR step are recoverable (deploy doesn't fail)
- `--no-pr` mode works for users without a remote
