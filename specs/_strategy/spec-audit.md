## Tag legend

- **KEEP** — spec is still correct under the new positioning; no changes needed.
- **REVISE** — spec is mostly right but needs targeted edits (a section here, an example there).
- **REWRITE** — spec's premise is broken by the new positioning; rewrite from scratch.
- **DELETE** — spec is no longer applicable; archive or remove.
- **HISTORICAL** — for shipped-code specs (M1, M1.1): the code exists, the spec describes what shipped. Don't rewrite the spec, but flag if the underlying *code* needs revision.

## Open questions — resolutions (2026-05-14)

1. **"Deploy" in a dlt world.** **Resolved.** `dlt deploy` is just code-generation scaffolding for external orchestrators (GitHub Actions, Airflow Composer); it doesn't push anything. dlt has no pre-provisioning step — destination schemas/tables get created lazily on first `pipeline.run()`. **Carve's `deploy` stays a pure Carve concept** (git-based promotion, PR-based for OSS, push-button for hosted). **P1-06 and P1.1-03 are now DELETE** rather than REVISE-or-DELETE — there's no pre-flight DDL phase to spec.

2. **dlt config files vs. Carve `destination.toml`.** **Resolved.** Adopt dlt's `.dlt/secrets.toml` and `.dlt/config.toml` directly. Drop Carve's `destination.toml`. The build agent writes dlt-shaped config files, not Carve-specific ones. Cascades: **P1.1-02 becomes DELETE-or-rewrite-as-thin-wrapper.**

3. **Env var convention.** **Resolved.** Adopt dlt's convention (`DESTINATION__SNOWFLAKE__DATASET_NAME`, etc.). Carve's `SCHEMA` env-var work in the working tree gets remapped to whatever dlt expects.

4. **Single combined process vs. separate scheduler/worker processes.** **Still open** — see *Still open* below.

5. **Curated dlt source library location.** **Resolved.** Lives at `src/carve/sources/`. Net-new directory under the Python package.

6. **EL agent prompt content.** **Resolved.** Single prompt that knows the dlt API. Rewrite the two M2-03-inherited prompt files (`snowflake_destination.md`, `data_engineering.md`) as one consolidated dlt-aware prompt for the EL specialist.

7. **P2 runtime step types.** **Resolved.** P2 ships `dlt`, `dbt`, and `sql` step types together. The three-consumer slate forces the step abstraction to be more honest than designing around one consumer. `sql` is cheap to add given `dlt` and `dbt` both need a destination connection — it's "execute this SQL file using that connection." Deferred to P3+: `shell`, `http`, `python`, `agent`, `approval`.

8. **M1 SQLite → Postgres migration.** **Resolved.** A one-shot `carve migrate-state` tool is needed (not "wipe and start over"). Repo inspection confirms: the state-store models are already cross-backend by design, but there are 6 Alembic migrations on SQLite (`0001_baseline` through `0006_recovery_chains`). Migration tool spec is its own small piece of work.

### Second-round questions

9. **Snowflake-first vs. destination-agnostic.** **Resolved.** Users can target non-Snowflake destinations from the first cut. Tests get authored and run when the user creates their EL pipeline — Carve maintainers don't pre-validate every dlt adapter.

10. **Local static HTML UI lineage scope.** **Resolved.** The initial UI is run history + logs only; lineage view is deferred. Keeps scope tight and avoids the cross-cutting "dlt resource → destination table → dbt source" correlation work.

4. **Worker process topology and default concurrency.** **Resolved.** Hybrid model: `carve serve` runs scheduler + N in-process workers (default `N=1`, serial); `carve serve --workers N` or `runtime.toml` setting raises in-process concurrency; `carve worker` runs as a separate process for cross-machine scale-out, all coordinating via the shared Postgres queue. OSS default behavior is one job at a time — predictable, easy to reason about, scales via one config knob when needed.
