# OSS packaging: bundled docker-compose for Postgres + external-Postgres option

> Delivers the first-run UX implied by [positioning #12](../_strategy/2026-05-positioning.md), and [ARCHITECTURE §5.7](../ARCHITECTURE.md): one-command Postgres for self-hosters, with a clean override for users who already operate Postgres.

## Status

- **Status:** Drafting
- **Depends on:** [state-store](./state-store.md) (the engine has to accept the connection string this spec produces)
- **Blocks:** [init](./init.md) (init scaffolds the compose file produced here)

## Goal

Make Carve's first-run experience one command for a self-hoster who doesn't already operate Postgres, while keeping a clean path for users who do. Concretely:

- `carve init` in a fresh project produces a `docker-compose.yml` and `.env` template that together let the user run `docker compose up -d` and immediately have a working Postgres instance on localhost
- `carve init --external-postgres "postgresql://..."` skips the compose-file scaffolding and points Carve at the user's existing Postgres
- The OSS distribution itself (the `carve` Python package and CLI) installs via `pip install carve` (or `pipx`/`uv tool install`); Carve does **not** require running its own code inside a container

The compose bundle is Postgres-only. Carve's API server, scheduler, and worker(s) run as a normal Python process via `carve serve` — not in a container. This keeps the install model simple ("it's a Python CLI tool that needs a database") and avoids the complexity of mounting the user's project directory into a container that runs dlt + dbt subprocesses.

## Out of scope

- Bringing up Carve itself in a container. Carve runs natively via `carve serve`. A user who wants to put Carve in their own container (e.g., systemd-managed or in a Kubernetes pod) can do so, but Carve doesn't ship its own image. (Note: the hosted product runs Carve in containers, but that's the hosted product's concern, not OSS packaging.)
- Helm charts, Kustomize overlays, or K8s manifests. Per [ARCHITECTURE §15](../ARCHITECTURE.md), we don't ship a K8s-native abstraction.
- Production-hardening the bundled Postgres (replication, backups, encrypted volumes, tuning). The bundle is for dev/test/single-team-self-host. Production users override with managed Postgres.
- The actual `carve init` UX (interactive prompts, brownfield detection, dlt/dbt scaffolding) — that's [init](./init.md). This spec only ships the compose template and the boundary in init code that says "drop this template in unless `--external-postgres` was passed."

## Behavior

### Install paths

Carve is a Python package. Supported install paths:

- `pip install carve` (into a user-managed venv)
- `pipx install carve` (isolated, recommended for end users who don't manage Python envs themselves)
- `uv tool install carve` (modern recommended path)

The CLI entry point is `carve` (Typer-based; see [ask](./ask.md) and the rest of the surface for what's exposed).

After install, the user has a `carve` binary on PATH. No additional setup is needed until they run `carve init` in a project.

### The bundled compose file

`carve init` (unless `--external-postgres` is passed) renders `docker-compose.yml` at the project root:

```yaml
# Generated by `carve init` at {timestamp}. Edit freely; carve will not regenerate this file
# on subsequent init runs unless you delete it first.
services:
  carve-postgres:
    image: postgres:16
    container_name: carve-postgres-{project_slug}
    environment:
      POSTGRES_USER: carve
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-carve}
      POSTGRES_DB: carve
    volumes:
      - carve-postgres-data:/var/lib/postgresql/data
    ports:
      # Bound to localhost only; never exposed to the network.
      - "127.0.0.1:${CARVE_POSTGRES_PORT:-5432}:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U carve -d carve"]
      interval: 5s
      timeout: 5s
      retries: 10
    restart: unless-stopped

volumes:
  carve-postgres-data:
    name: carve-postgres-data-{project_slug}
```

Where `{project_slug}` is a slug derived from the project directory name (so a user with multiple Carve projects gets multiple isolated Postgres instances rather than name collisions).

The corresponding `.env.example` gains:

```bash
# State store connection. Defaults match the bundled docker-compose.yml.
# Override if you're using an external Postgres (managed RDS, Cloud SQL, Supabase, etc.).
DATABASE_URL=postgresql+psycopg://carve:carve@127.0.0.1:5432/carve

# Set to override the bundled Postgres password. Default is "carve" (fine for local dev).
# POSTGRES_PASSWORD=
# Set to override the bundled Postgres host port. Default is 5432.
# CARVE_POSTGRES_PORT=
```

### The `--external-postgres` flag

```bash
carve init --external-postgres "postgresql+psycopg://user:pass@host:5432/db"
```

Behavior:

- **Validates** the connection string format (must start with `postgresql+psycopg://` or `postgresql://` — the latter gets a friendly note that we'll add `+psycopg` for clarity)
- **Connects** to confirm Postgres is reachable and that the user has CREATE TABLE privileges (Carve needs to run Alembic migrations)
- **Does not** scaffold `docker-compose.yml` (no compose file is written)
- **Writes a commented placeholder** (`# DATABASE_URL=postgresql+psycopg://USER:PASSWORD@HOST:5432/DB`) to `.env.example` — **never the real, password-bearing string** (`.env.example` is conventionally committed to git, and the connection string is a secret, per [init](./init.md)'s "init never writes real secrets" rule) — and **prints** the real `DATABASE_URL` line for the user to paste into their gitignored `.env`
- **Prints** a note: "External Postgres detected; bundled docker-compose not generated. Carve will not manage your Postgres instance lifecycle — backups, upgrades, and tuning are your responsibility."

If `--external-postgres` is omitted, the bundled path is taken (compose file rendered).

### State-store initialization (migration timing)

`carve init` brings the schema to head (`initialize_database` → `alembic upgrade head`) **when Postgres is reachable**:

- **External** Postgres is already running, so init migrates immediately; an unreachable database, or a connecting user lacking CREATE TABLE, is a hard error (exit 3). The engine is built from the provided URL **directly** (not via `resolve_state_store_url`), so a stray `DATABASE_URL` env can't redirect it.
- **Bundled** Postgres usually isn't up at init time (the compose file was only just rendered), so init does **not** fail: it prints a next-step ("`docker compose up -d`, then `carve serve`") and the schema is brought to head once Postgres is reachable.

### Idempotency: re-running `carve init`

If a `docker-compose.yml` already exists in the project root, `carve init` leaves it alone and prints a message: "Found existing docker-compose.yml; not overwriting. Delete it first if you want a fresh template." This protects users who have customized their compose file.

Same for `.env`: init never writes or overwrites it. And if `.env.example` already exists, init leaves it alone too — the state-store block is written only on first scaffold; delete it first for a fresh template.

### Lifecycle commands

After `carve init`, the user runs:

```bash
docker compose up -d           # brings up Postgres in background
carve serve                    # brings up the API/scheduler/worker(s)
# ...in another terminal:
carve plan "ingest the Stripe charges API"
```

To stop:

```bash
# Ctrl-C the `carve serve` process
docker compose stop            # leaves volume in place
# or:
docker compose down            # removes containers but keeps the named volume (data preserved)
# or:
docker compose down -v         # also removes the volume (DATA LOSS)
```

Carve doesn't wrap any of these — they're standard Docker Compose commands. The docs make this clear.

### Backup / restore

`docs/first-run-with-bundled-postgres.md` includes a short "production considerations" section noting:

- The bundled Postgres has no automatic backup; for production-ish use, either run `pg_dump` on a cron or migrate to managed Postgres
- The named volume persists across `docker compose down` and re-up; only `down -v` destroys data
- For real production, use `--external-postgres` against a managed instance

## Tests

- **Unit**: the compose template renders with expected placeholders substituted, given a project name and config
- **Unit**: `carve init --external-postgres <url>` rejects malformed connection strings
- **Integration**: `carve init` in a fresh tempdir produces a valid `docker-compose.yml`; `docker compose up -d` brings Postgres up; `pg_isready` succeeds; `carve init`'s state-store initialization (`initialize_database` → `alembic upgrade head`) connects to the bundled Postgres and brings the schema to head. *(The full `carve serve` runtime is exercised in [spec 07](./runtime.md), not here — `serve` is still a stub at this spec's build time.)*
- **Integration**: `carve init --external-postgres <url>` against a testcontainers-provided Postgres skips compose scaffolding and successfully connects + migrates
- **Integration**: re-running `carve init` in a project that already has `docker-compose.yml` leaves the file untouched and exits cleanly
- **Integration**: two `carve init`s in two different parent directories produce two non-colliding Postgres containers (verifies `{project_slug}` substitution)

## Acceptance

- A user who has Docker installed but no Postgres can run, in a fresh project directory, `carve init` → `docker compose up -d` in under 2 minutes total; the schema is brought to head once Postgres is up (init defers the migration with a next-step if Postgres isn't running yet, then `carve serve` brings it to head). (Bringing up the API/scheduler/worker via `carve serve` is verified in [runtime](./runtime.md).)
- A user who has managed Postgres (RDS, Cloud SQL, Supabase, local install) can run `carve init --external-postgres "postgresql+psycopg://..."`, and Carve validates + migrates the schema against it, skipping Docker entirely. (The `carve serve` runtime lands in [spec 07](./runtime.md).)
- The bundled Postgres is bound to `127.0.0.1` only (verifiable via `ss -tlnp` or `netstat`); never accessible from other machines on the network
- Multiple Carve projects on the same machine don't collide on container or volume names
- `docs/installation.md`, `docs/first-run-with-bundled-postgres.md`, and `docs/first-run-with-external-postgres.md` each walk a user through to a successful `carve plan` in under 15 minutes

## Design notes

- **Why Postgres-only in the compose bundle (not Carve too)?** Three reasons. (1) Carve is a Python CLI tool that the user installs with `pip`/`pipx`/`uv tool`; running it natively avoids the complexity of mounting the user's project directory into a container that runs dlt/dbt subprocesses against it. (2) Debugging is dramatically easier when Carve runs natively — `carve serve` outputs go straight to the terminal; no `docker logs` required. (3) Hosted is the right answer for users who want a fully-managed runtime; building a polished Carve-in-Docker OSS image would compete with the hosted product without the operational benefits.
- **Why `postgres:16`?** Latest LTS major version at authoring time, stable, widely available. Pin to a major version (not `latest`) so existing installs don't get surprise updates. Bump in a future release when 17 stabilizes.
- **Why `127.0.0.1` binding?** Security default; the bundled Postgres should never be exposed to the network. Users who actually need network access (e.g., running Postgres on a different machine than `carve serve`) should use `--external-postgres` and put their Postgres behind real auth.
- **Why slug-prefixed container names?** Lets the user run multiple Carve projects on the same host without `container_name` collisions. Docker Compose's default project-name behavior also handles this, but explicit naming is friendlier for `docker ps`.
- **Why not Helm / K8s manifests?** Per [ARCHITECTURE §15](../ARCHITECTURE.md), K8s is supported in hosted but not a Carve-native abstraction in OSS. A community-contributed Helm chart can land later if there's demand; it's not a blocker.

## Open questions

> Tagged per the scheme in spec 01.

- **Postgres 16 vs 15.** *Implementation default.* Go with 16; rollback to 15 if any test infrastructure incompatibility surfaces. Document the supported version in `docs/installation.md`.
- **Whether to ship a Makefile or `just` recipes for the common Docker commands.** *Implementation default.* Skip for now; the three commands (`docker compose up -d`, `docker compose down`, `carve serve`) are short and well-known. Revisit if user feedback says it's friction.
- **Behavior when Docker is not installed and `--external-postgres` was not passed.** *Implementation default.* `carve init` detects the absence of `docker compose` on PATH and prints a friendly message: "Docker not detected. Either install Docker Desktop / `docker compose`, or re-run with `--external-postgres "postgresql+psycopg://..."` pointing at your existing Postgres." Exits with code 3 (config error).
- **Whether the bundled compose includes any other services.** *Implementation default.* No — Postgres only. Adding services (Redis for the hosted event bus, an init container for first-run migrations, a healthcheck companion) accumulates complexity without OSS benefit. Hosted bundles its own thing.
