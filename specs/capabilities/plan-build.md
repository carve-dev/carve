# Plan / Build: the change lifecycle and the Plan & Build entities

> **Updated during implementation (2026-06-25) — lean plan/build ([ADR](../_strategy/2026-06-lean-plan-build.md)):** the `terraform plan`/`apply` framing below is **softened**. "Plan/build" bundles two separable things whose ownership now splits: the **interactive "review before code" gate is the [harness](./harness.md)'s** (`plan`-mode + the permission gate — the engineer's *design capacity*; plan-build does **not** duplicate it), and the **durable, reviewable, auditable record of an authored change is git + [deploy](./deploy.md)** (the diff/PR — Carve does not reinvent that as a Plan-artifact store). The **Plan *entity* is reserved for what neither provides:** (a) **cost/runtime/impact estimated *up front*** (before any code is authored), (b) the **config-hash drift gate** (shipped), and (c) the **non-conversational "plan now, build later" surface** (REST/MCP/CI/schedule). Heavyweight pieces (refine chains/`--refine`, plan expiry, `plan-and-build`, heavyweight synthesis-into-one-Plan) are **kept only where a use-case demands** — the shipped M1.1 + Unit-1 versions stay; they are not expanded speculatively. Nothing shipped is reverted.

> **The durable design of record for Carve's central lifecycle** — `plan → build → (run / deploy)`. A **Plan** is a durable artifact carrying its **up-front estimate** (task graph, proposed file diffs, **cost + runtime estimate**, impact analysis) and its **config hash** (the drift anchor); a **Build** materializes an approved Plan into files (a `manifest`, `current_build_id`). This capability owns the **Plan/Build entities, their persistence, the `plan`/`build`/`plan-and-build` verbs, `--refine` chaining, and the config-hash drift check.** The *interactive review-before-code gate* is the [harness](./harness.md)'s (`plan`-mode/design-capacity); the *durable diff/review of an authored change* is git + [deploy](./deploy.md)'s; this spec owns the **entity (the up-front estimate + drift gate + non-conversational surface), the synthesis of subagent results into that estimate, the lifecycle, and the safety net.** *Phasing annotation:* the core (plan/build/refine/run separation) **shipped in M1.1**; this spec is its durable design of record and the home for its forward evolution (the dbt-build path, richer impact analysis).

## Status

- **Status:** Drafting (durable design for shipped + evolving behavior)
- **Depends on:** [harness](./harness.md) (the orchestrator that delegates + returns usage/cost per subagent), [state-store](./state-store.md) (the `plans`/`builds` tables + the `.carve/plans/<id>.json` artifacts), [layout](./layout.md) (the `config_hash` over `carve.toml` + component refs), [pipelines](./pipelines.md) / [dlt-engineer](./dlt-engineer.md) / [dbt-engineer](./dbt-engineer.md) (the subagents whose authored diffs a Plan composes; the Build manifest they fill).
- **Used by:** [deploy](./deploy.md) (consumes the Build manifest + `config_hash`), [rest-api](./rest-api.md) / [mcp-server](./mcp-server.md) (expose `/plans` + `/builds`; this spec owns the services behind those routers), [runtime](./runtime.md) (a run executes a pipeline materialized by a Build), [recovery](./recovery.md) (a fix surfaces as a proposed Plan).
- **Lineage:** the plan/build/run separation shipped in **M1.1** ([`../milestone-1.1-followups/06-plan-build-run-separation.md`](../milestone-1.1-followups/06-plan-build-run-separation.md)). This capability spec is the durable design that was missing under the three-tier model.

## Goal

Own the change lifecycle as a first-class, reviewable thing: **`carve plan` produces a durable, inspectable Plan** (no files written); **`carve build` materializes it** (files written, not deployed); the two are decoupled so a Plan can be reviewed, refined, diffed, and built later — with a **config-hash drift check** that refuses to build a Plan against config that moved underneath it.

## Out of scope

- **Authoring** the contents of a plan — the [dlt-engineer](./dlt-engineer.md) / [dbt-engineer](./dbt-engineer.md) / [pipeline engineer](./pipelines.md) write the code; this spec composes their results into a Plan and records the Build.
- **The orchestrator loop / delegation mechanics** — [harness](./harness.md). This spec consumes the harness's per-subagent `DelegationResult` (usage/cost) to roll up the Plan's cost.
- **Running** a built pipeline ([runtime](./runtime.md)) and **promoting** it ([deploy](./deploy.md)).
- **The raw table DDL** — [state-store](./state-store.md) owns the `plans`/`builds` table creation; this spec owns their *semantics*.

## Behavior

### The Plan entity

> **Updated during implementation (2026-06-25) — lean plan/build ([ADR](../_strategy/2026-06-lean-plan-build.md)):** the Plan entity is **reserved for what the harness's gate and git/deploy don't provide:** (a) the **up-front cost/runtime/impact estimate** (before any code is authored), (b) the **config-hash drift gate** (below), and (c) the **non-conversational "plan now, build later" surface** (REST/MCP/CI/schedule — where there's no live conversation to be the gate). Most interactive, conversational changes flow through the harness's `plan`-mode gate and land as a git diff/PR; they do **not** need a formal Plan entity. **`expires_at`/plan expiry is kept where a use-case demands it** — the shipped 24h-default behavior stays; it is not elaborated speculatively ([ADR](../_strategy/2026-06-lean-plan-build.md) Consequences).

A Plan is a durable artifact persisted to `.carve/plans/<plan_id>.json` **and** indexed in the `plans` table:

- **Goal** (the user's request), **task graph** (the steps the orchestrator decomposed it into), **file diffs** (what each subagent would write), **expected effects / impact analysis** (which pipelines/tables/downstream are affected — leans on [lineage](./lineage.md) investigation).
- **Cost + runtime estimate.** The Plan surfaces the **exact LLM cost** (known precisely — summed from each subagent's `DelegationResult.usage`/`cost_usd`, per [harness](./harness.md)) and an **estimated runtime** (first run vs. subsequent — e.g. "~25 min first load / <1 min incremental"), composed from the engineers' `expected_outputs`. **No warehouse-dollar estimate** (we can't know it precisely — UC1 resolved).
- **`config_hash`** over `carve.toml` + the resolved component refs at plan time — the drift anchor.
- **`parent_plan_id`** for refinement chains; **`carve_version`**; **`status`**; **`expires_at`** (Plans expire, default 24h).

### Plan synthesis (the up-front estimate)

> **Updated during implementation (2026-06-25) — lean plan/build ([ADR](../_strategy/2026-06-lean-plan-build.md)):** the **interactive human-in-the-loop gate is the harness's**, not a duplicate mechanism here. The engineer's **design capacity** (running at `plan`/read authority, proposing without writing, the permission gate clamping writes off) *is* that gate — Carve does not re-implement "propose then approve" as a parallel entity-driven flow. What this synthesis owns is the **up-front estimate** (cost/runtime/impact rolled up *before* any code is authored) — the value git/the diff can't give until after authoring. **`--refine` chaining is kept where a use-case demands it** — the shipped child-Plan behavior stays; richer refine chains are built only when a concrete workflow needs them, not because the terraform analogy suggested them ([ADR](../_strategy/2026-06-lean-plan-build.md) Consequences).

`carve plan "<goal>"` runs the [orchestrator](./harness.md): classify → delegate to the right subagents **in a design capacity** (each runs at **plan/read authority** — using its read tools + domain expertise to *propose* what it would build: the file manifest it would author, its chosen strategy, dependency hints, and a cost/runtime estimate — **without authoring or verifying the component code; that is build-time**) → **synthesize the up-front estimate**: merge the proposed file manifests, roll up the exact LLM cost, compose the runtime estimate, and render the impact analysis. The estimate is returned to the surface (CLI/chat/REST) for review; **no code is written or modified until the human accepts and runs `carve build`** — the plan→build split rides the harness's `plan`-mode gate (a plan may write an incidental scaffold file if a design genuinely needs one, but never the component code). The **verified diff** (the engineer authoring its slice + running it to green via the verification loop) is therefore a **build-time** behavior; at plan time the engineer returns a *proposed* diff/manifest, not an authored-and-verified one — and once authored, **the durable, reviewable record of that change is the git diff + the PR ([deploy](./deploy.md)), not a re-rendered Plan artifact**. `carve plan --refine <plan_id> "<feedback>"` produces a child Plan (`parent_plan_id` chain); `carve plan --pipeline <name> "<change>"` scopes to an existing pipeline.

### The Build entity + materialization

`carve build <plan_id>` materializes an approved Plan's task graph into files (dlt code, `pipelines/<name>.toml`, dbt models when the [dbt-engineer](./dbt-engineer.md) is in play), emits the `post_build` hook ([extensibility](./extensibility.md)), and records a **`Build`** row with a **`manifest_json`** listing every file written. A pipeline's **`current_build_id`** points at its most recent successful Build (what [deploy](./deploy.md) promotes, what [runtime](./runtime.md) runs). Build is **idempotent** (re-building the same Plan against unchanged config is a no-op). `carve plan-and-build "<goal>"` is the one-shot convenience (plan, then immediately build — for trusted/CI flows); per lean plan/build ([ADR](../_strategy/2026-06-lean-plan-build.md)) it is **kept where a use-case demands it** — the shipped thin verb stays, but a heavyweight `plan-and-build` synthesis path is not built speculatively.

> **Updated during implementation (2026-06-26):** the `post_build` **emitter is now LIVE and owned by this capability's builder.** `POST_BUILD` has been lifted out of `DEFERRED_EMITTER_EVENTS` into `EMITTED_EVENTS` (`core/hooks/events.py`); the builder owns + fires it via `build_post_build_hook` (`core/hooks/wiring.py`, the lifecycle analogue of `build_tool_hooks`) wrapped by `build_extensibility_post_build_hook` (`cli/orchestrator/extensibility_wiring.py`), fired in **both** build paths — `builder.build_plan` (the M1 path) and `_build_multi_engine` — **after** the `Build` row + `current_build_id` are durably recorded. The hook is **gated at `PermissionMode.BUILD`** (the same bash gate as tool hooks). The **post-commit fail-closed contract** is the one place a lifecycle hook diverges from a pre-action tool hook: the bash gate still **DENIES dangerous commands** (`$()`/`;`/`&&`/dangerous programs) at the command level (fail-closed), but a hook that **RAISES at fire time** (after passing the gate) is **surfaced/logged at error and the Build STANDS** — the run is **not** flipped to `failed` and the Build is **not** un-recorded (the materialization genuinely happened). A blocked or failed build, and the idempotent rebuild no-op, do **not** emit `post_build`. [Extensibility](./extensibility.md) ships only the *subscription* seam; the `HookRegistry` is deliberately **not** used in `src/`. See [DELIVERY.md](../DELIVERY.md) → *Current state* for the recorded ownership decision.

### Config-hash drift check (the safety net)

Every Plan carries the `config_hash` it was generated against. **Build refuses to run a Plan whose `config_hash` no longer matches current config** (`carve.toml`/component refs moved since plan time) — exit `3`, with a clear "re-plan against current config" message. Deploy ([deploy](./deploy.md)) does the analogous pre-flight (exit `4` on target drift). This is the per-verb hash gate (ARCHITECTURE §7.6) that keeps "plan now, build later" safe.

## Tests

- **Unit (Plan entity):** a plan persists to `.carve/plans/<id>.json` + a `plans` row with task graph, file diffs, exact LLM cost, runtime estimate, `config_hash`, `expires_at`; an expired plan is rejected by build.
- **Unit (synthesis/cost):** the Plan's cost equals the sum of the subagents' `DelegationResult` usage; the runtime estimate composes from `expected_outputs`; **no warehouse-dollar figure** is emitted.
- **Unit (refine chain):** `plan --refine` sets `parent_plan_id`; the chain is walkable.
- **Integration (build materializes):** `carve build <plan_id>` writes exactly the Plan's file set, records a `Build` with `manifest_json`, updates `current_build_id`, emits `post_build` **after** the Build is recorded (post-commit: a raising `post_build` hook is logged and the Build stands, not rolled back; a blocked/failed build and the idempotent no-op do not emit it); re-building unchanged config is a no-op.
- **Integration (drift):** mutating `carve.toml` after plan then `carve build <plan_id>` fails with exit 3 (`config_hash` mismatch) and a re-plan message.

## Acceptance

- A Plan is a **durable, reviewable artifact** (task graph + file diffs + exact LLM cost + runtime estimate + impact analysis + `config_hash`), persisted to disk + the `plans` table, refinable via `--refine`, expiring by default.
- **`build` materializes a Plan into files** (recording a `Build` manifest + `current_build_id`), emits `post_build` (after the Build is durably recorded, gated at `BUILD`, post-commit fail-closed — a raising hook is surfaced but the Build stands), and is idempotent. *(The `post_build` emitter is this capability's responsibility against [extensibility](./extensibility.md)'s shipped subscription seam — see the §"The Build entity" callout.)*
- **The config-hash drift check** refuses to build a Plan against drifted config (exit 3).
- The `plan` / `build` / `plan-and-build` verbs + `/plans` + `/builds` services are owned here (the [rest-api](./rest-api.md) routers wire onto them).

## Design notes

> **Updated during implementation (2026-06-25) — lean plan/build ([ADR](../_strategy/2026-06-lean-plan-build.md)):** the "spine of every change" / "synthesis-into-one-reviewable-Plan" framing below is the heavyweight reading. Under lean plan/build the *interactive review spine* is the harness's gate and the *durable change record* is git/deploy; what justifies this dedicated capability is the **up-front estimate + the config-hash drift gate + the non-conversational surface** that neither of those provides. The synthesis this spec owns is the synthesis of subagent results into that **estimate**, not into a bespoke durable Plan-artifact that competes with the git diff.

- **Why a dedicated capability?** plan/build owns the entities, persistence, drift logic, refine chains, and verbs + a REST/MCP surface for the *up-front estimate and drift gate* — too central and cross-cutting to live as an annotation, yet it had no home (only the shipped M1.1 work order). The audit found rest-api and state-store both explicitly pointing ownership elsewhere. (The *interactive review-before-code* spine is the harness's `plan`-mode gate; the *durable diff/PR* is git/deploy's — see the lean plan/build [ADR](../_strategy/2026-06-lean-plan-build.md).)
- **Why synthesis lives here, not in harness.** The harness owns *how* the orchestrator delegates (the loop, the gate, `DelegationResult`). Assembling those results into the **up-front cost/runtime/impact estimate** is a *lifecycle* concern — the deliverable, not the mechanism. (Where a change is authored, the human reviews the resulting **diff/PR** via deploy, not a re-rendered Plan.)
- **Why exact LLM cost but no warehouse estimate.** Carve knows token spend precisely; warehouse compute depends on data volume/warehouse size Carve can't predict. Honesty over a fake number (UC1).
