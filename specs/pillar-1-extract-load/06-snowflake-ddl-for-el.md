# P1-06 — Snowflake DDL generation for EL destinations

**Milestone:** Pillar 1 — Extract & Load
**Estimated effort:** 0.5 day
**Dependencies:** M1-06 (Snowflake connector), P1-02 (plan/build lifecycle), P1-04 (extract-load agent), P1-05 (schema retrieval)
**Lineage:** Narrow subset of **M2-05** ([`specs/_archive/milestone-2-real-product/05-snowflake-agent.md`](../_archive/milestone-2-real-product/05-snowflake-agent.md)). Pillar 1 ships only the per-EL DDL emission portion (the "Per-pipeline output" section added during the SDLC discussion). The full Snowflake agent's broader scope — warehouse creation/sizing, role hierarchies, account-level RBAC, dynamic tables, streams, tasks — defers to **Pillar 2** or later.

## Purpose

Define the contract for the per-EL DDL companion file that the extract-load agent (P1-04) emits and that `carve el deploy`'s DDL-apply phase (P1-08) executes. The DDL file ensures the destination schema, table, and runtime-role grants exist before the EL script runs against the target — without the EL script itself having to do `CREATE` operations at run time (which would require elevated privileges on the runtime role).

This spec defines the file format and idempotency contract; the *agent* that produces it is P1-04, the *command* that applies it is `carve el deploy` (P1-08).

## File location and naming

```
targets/<active_target>/snowflake/<artifact_name>.sql
```

One DDL file per EL artifact. The file is committed to the repo (tracked in git, not gitignored). It carries through the deploy phase via `carve el deploy --from X --to Y` (P1-08): copying the EL artifact also copies its DDL file into the destination target's tree.

## Required contents

A Pillar 1 DDL file contains, in order:

1. **`CREATE SCHEMA IF NOT EXISTS`** for the destination schema (skipped when the schema already exists in the user's standard layout — agent uses P1-05's `table_exists`-style checks at plan time to decide).
2. **`CREATE TABLE IF NOT EXISTS`** for the destination table, with columns matching the EL script's writes. Column types come from the design's `columns` list. Primary key from `destination.primary_key` if set.
3. **`GRANT SELECT, INSERT, UPDATE, DELETE ON TABLE <fqn> TO ROLE <runtime_role>`** for the runtime role (the role under `[snowflake.<target>]` in `carve/connections.toml`). The deploy role applies this; the runtime role then has the privileges needed for `carve el run`.
4. (Optional) **`CREATE STAGE IF NOT EXISTS`** when the script uses an internal stage for `COPY INTO` loads.
5. (Optional) **`CREATE FILE FORMAT IF NOT EXISTS`** when the script reads from a custom file format.

Sections are separated by blank lines + a single-line comment header (`-- === Schema ===`, `-- === Table ===`, `-- === Grants ===`) for readability when reviewers eyeball the diff during PR review at deploy time.

### Sample

```sql
-- Auto-generated by carve build for pipeline iowa_liquor_sales (target: dev).
-- Applied by `carve el deploy iowa_liquor_sales --from <X> --to dev` during its DDL-apply phase.
-- All statements are idempotent; re-running is safe.

-- === Schema ===
CREATE SCHEMA IF NOT EXISTS ANALYTICS_DEV.RAW;

-- === Table ===
CREATE TABLE IF NOT EXISTS ANALYTICS_DEV.RAW.IOWA_LIQUOR_SALES (
    INVOICE_LINE_NO  VARCHAR(50)   NOT NULL,
    DATE             DATE,
    STORE_NUMBER     VARCHAR(20),
    ITEM_DESCRIPTION VARCHAR(255),
    BOTTLES_SOLD     NUMBER(10,0),
    SALE_DOLLARS     NUMBER(12,2),
    LOCATION         VARIANT,           -- nested dict from Socrata
    _LOADED_AT       TIMESTAMP_NTZ DEFAULT CURRENT_TIMESTAMP(),
    PRIMARY KEY (INVOICE_LINE_NO)
);

-- === Grants ===
GRANT SELECT, INSERT, UPDATE, DELETE ON TABLE ANALYTICS_DEV.RAW.IOWA_LIQUOR_SALES
    TO ROLE TRANSFORMER_DEV;
```

The file references concrete database and role values (resolved by the agent from the active target's `[snowflake.<target>]` section at build time). It does **not** use `${ENV_VAR}` substitution — the file is committed as a snapshot of "what this build expects in this target." If the user changes `connections.toml` after the build, they re-build to regenerate the DDL.

## Idempotency contract

Pillar 1 enforces a strict rule: **every statement in the DDL file must be safe to re-run any number of times** with no side effects beyond the first successful application. Destructive intent (drop/replace/rename) is allowed *only when the destructive verb has an idempotent form Snowflake supports* AND the user has approved the destructive plan at plan time (see "Destructive intent: prompted at plan time" below).

Always allowed (additive, idempotent):
- `CREATE … IF NOT EXISTS` (schema, table, stage, file format)
- `GRANT …` on objects (Snowflake grants are idempotent)
- `ALTER TABLE … ADD COLUMN IF NOT EXISTS`

Allowed when the user approved the destructive intent at plan time:
- `DROP TABLE IF EXISTS` / `DROP COLUMN IF EXISTS` (idempotent — second run is a no-op)
- `DROP SCHEMA IF EXISTS … RESTRICT` (only when explicitly approved; cascade is forbidden)
- A combined `DROP … IF EXISTS; CREATE … IF NOT EXISTS;` pair when the user wanted "replace this object"

Always forbidden:
- `CREATE OR REPLACE …` (overwrites silently; if you mean replace, emit `DROP IF EXISTS` + `CREATE IF NOT EXISTS` — same effect, more legible in PR review)
- `RENAME …` (Snowflake's `ALTER TABLE … RENAME` is non-idempotent and has no `IF EXISTS` form; if a user wants to rename, the agent emits `ALTER TABLE … ADD COLUMN new`, `UPDATE … SET new = old`, then proposes a follow-up plan to drop the old column once the user has migrated downstream consumers)
- `ALTER COLUMN … SET DATA TYPE …` (lossy if data is already there)
- Non-idempotent DML embedded in DDL (e.g. `INSERT INTO migrations_log`)

The agent enforces this in its prompt's hard rules (P1-04). `carve el deploy`'s DDL-apply phase (P1-08) does *not* re-validate idempotency — it trusts the file. The PR review (in whatever git-workflow the user wraps `carve el deploy` with) is the human-driven check.

## Destructive intent: prompted at plan time

When the plan agent (M1.1-06) detects that a goal would require destructive changes against the active target — dropping a column, replacing a table, etc. — it **does not silently emit the destructive DDL**. Instead, the destructive intent surfaces in the design (and rendered plan) so the user explicitly chooses before `carve build` runs.

Mechanism: the plan agent's `submit_plan(design)` payload includes:

- A `tradeoffs` array (already part of the M1.1-06 design schema). Destructive intent surfaces here as: `"Will drop existing column `foo` (5 rows present); user data lost on apply."`
- An `open_questions` array (already part of the design schema). When the agent isn't certain the destructive path is what the user wants, it adds an open question: `"Existing column `foo` present. Drop it as part of replacement, or keep both columns? Refine the plan to choose."`

The CLI's plan rendering (M1.1-06's `render_plan_to_console`) prints these prominently. The user has three responses:

1. **Approve as-is** — run `carve build <plan_id>`. The destructive DDL gets emitted into `targets/<active>/snowflake/<artifact>.sql` and reviewed again at PR time before being applied to a non-dev target.
2. **Refine** — `carve plan --refine <plan_id> "keep the old column; add a new one alongside it"` produces a sibling plan that does additive-only DDL.
3. **Abandon** — discard the plan; do nothing.

The plan-time prompt is the **first gate**. The PR review at deploy time (P1-08) is the **second gate** — a human reviewer eyeballing `targets/prod/snowflake/<artifact>.sql` sees the `DROP TABLE IF EXISTS` line in plain SQL before approving the merge.

There is no interactive prompt mid-plan-generation (no curses-style "press y/n"). The render-then-refine loop is the OSS-friendly equivalent: stateless, scriptable, gits cleanly.

For destructive operations Snowflake doesn't support an idempotent form for (notably bare `RENAME`), the agent refuses via `submit_step(error=True, summary="Snowflake doesn't support idempotent RENAME; please drop and re-create, or hand-edit the target.")`. This is rare and the user can handle these as one-off manual SQL.

## Build manifest entry

P1-02's Build row stores a `manifest_json` listing every file the build produced. The DDL file is included:

```json
{
  "files": [
    "el/iowa_liquor_sales/main.py",
    "el/iowa_liquor_sales/requirements.txt",
    "snowflake/iowa_liquor_sales.sql"
  ]
}
```

Paths are relative to the target directory (`targets/<active>/`). `carve el deploy` (P1-08) reads the manifest to know which files to copy from `--from` to `--to` and which DDL file to apply during its DDL-apply phase.

## `carve el deploy`'s DDL-apply phase (preview of P1-08)

This spec defines the file; P1-08 defines the command and its DDL-apply phase:

```
carve el deploy <artifact> --from <X> --to <target>
```

During its DDL-apply phase (between file-copy and smoke-verify), `carve el deploy`:

1. Reads `targets/<target>/snowflake/<artifact>.sql` (just-copied from `<X>`).
2. Connects with the **deploy role** (`[snowflake.<target>_deploy]` in connections.toml — by convention; P1-08 spec details).
3. Executes each statement in the DDL file in order.
4. Reports applied / no-op statements; on any failure, hands off to the recovery agent (P1-09) within budget.
5. On budget exhaust: exits non-zero with the failing statement and the recovery agent's diagnosis.

Re-running `carve el deploy` on a target where the DDL was already applied is a no-op for the DDL-apply phase (idempotent). This is the core safety property — users running the workflow twice don't damage anything.

## Modify-existing-pipeline path

When the agent is invoked with `action="modify_extractor"` (existing pipeline gets a code change), it re-emits the DDL file fresh. The content typically diverges only when the design changes (e.g., a new column added to the script). The fresh DDL still uses `CREATE … IF NOT EXISTS` and `GRANT …` — applying it against an already-provisioned target is a no-op for unchanged statements and an idempotent additive change for new ones.

For schema *evolution* (column added to an existing table), Pillar 1 supports the additive case via `ALTER TABLE … ADD COLUMN IF NOT EXISTS …` if the agent emits it. Snowflake supports this; the agent has a sub-section in the Snowflake destination skill (P1-04) covering it. Non-additive schema evolution (column type change, drop, rename) is out of scope — see "Idempotency contract" above.

## Implementation

The DDL emission *implementation* lives in the extract-load agent (P1-04). The agent writes SQL directly — no programmatic-DDL-builder layer in Python. Reasoning: the DDL file is committed to the repo and reviewed at deploy time; SQL the agent wrote is exactly what reviewers see, and there's no helper-layer to debug when output looks wrong. The Snowflake destination skill (P1-04 / `src/carve/skills/snowflake_destination.md`) carries an extensive sub-section with DDL emission patterns the agent can copy from.

A regression test (`tests/core/agents/test_extract_load_agent.py::test_emitted_ddl_parses`) validates the agent's emitted SQL parses via `sqlparse` (or similar) and contains the expected statement types. This catches malformed output early.

The DDL file's *application* (running it against Snowflake) lives in P1-08's `carve el deploy` flow — DDL-apply phase, deploy-role connection, error handling.

## Tests

- `test_ddl_emitted_for_new_artifact` — `carve build` for a new artifact produces the DDL file with the documented sections.
- `test_emitted_ddl_parses` — emitted SQL parses via `sqlparse`; statement types match expected (CREATE SCHEMA, CREATE TABLE, GRANT).
- `test_ddl_idempotent_create_table_if_not_exists` — every CREATE/GRANT statement uses idempotent form (`IF NOT EXISTS`, `GRANT`).
- `test_ddl_never_uses_create_or_replace` — agent never emits `CREATE OR REPLACE`; replacement intent always emits `DROP IF EXISTS` + `CREATE IF NOT EXISTS` pair.
- `test_ddl_never_uses_bare_rename` — agent never emits bare `ALTER TABLE … RENAME` (no idempotent form). Test fixture: a goal that implies a rename → agent surfaces `submit_step(error=True)` or a refactor proposal in the design.
- `test_ddl_destructive_intent_surfaces_in_tradeoffs` — fixture: existing column `foo` present, goal implies replacement. Agent's design's `tradeoffs` array includes `"Will drop existing column foo …"` and the emitted DDL includes `DROP COLUMN IF EXISTS foo`.
- `test_ddl_modify_path_emits_alter_add_column` — adding a column to an existing artifact's design produces an `ALTER TABLE … ADD COLUMN IF NOT EXISTS` statement.
- `test_ddl_grants_runtime_role` — emitted GRANT references the runtime role from the active target's `[snowflake.<target>]` section.
- `test_build_manifest_includes_ddl_file` — Build.manifest_json lists `snowflake/<artifact>.sql` as one of the files.

## Acceptance criteria

- The extract-load agent emits a valid DDL file at `targets/<active>/snowflake/<artifact>.sql` for every build.
- The file contains schema `CREATE IF NOT EXISTS` + table `CREATE IF NOT EXISTS` + grants for the runtime role.
- Re-running `carve el deploy` on an unchanged source is a no-op for the DDL-apply phase.
- Adding a column to an existing artifact emits `ALTER TABLE … ADD COLUMN IF NOT EXISTS` rather than `CREATE OR REPLACE`.
- Destructive intent (drop column, replace table) surfaces in the plan's `tradeoffs` and (when ambiguous) `open_questions` so the user reviews and approves at plan time before the destructive DDL gets emitted into the build.
- Approved destructive DDL uses `DROP … IF EXISTS` form (never bare `DROP`); replacement intent emits `DROP IF EXISTS; CREATE IF NOT EXISTS;` rather than `CREATE OR REPLACE`.
- Bare `RENAME` is never emitted (no idempotent form); rename-shaped goals get a refactor proposal in the design or `submit_step(error=True)`.
- The Build's manifest_json lists the DDL file alongside `main.py` and `requirements.txt`.
- The emitted SQL parses (regression test); two human gates exist: plan-time render-and-approve, and PR review at deploy time.
- `ruff` + `mypy --strict` + `pytest` stay green.

## Out of scope

- Warehouse creation, role hierarchies, account-level RBAC — full Snowflake agent territory; defers to Pillar 2 or later.
- Migration files (`pipelines/<name>/migrations/NNN_slug.sql`) for non-idempotent operations — that pattern was in the parked M2-14 deploy proposal but is deferred entirely from Pillar 1. Idempotent additive DDL covers the v0.1 cases.
- Stages, file formats, dynamic tables, streams, tasks beyond the basic optional `CREATE STAGE IF NOT EXISTS` and `CREATE FILE FORMAT IF NOT EXISTS` listed above.
- Drop / rename / schema-evolution operations that aren't safely idempotent — surfaced via `submit_step(error=True)` in P1-04.
- Cross-database moves (table from one database to another). Out of scope.
- Network policies, masking policies, row-access policies — Snowflake-security territory that doesn't belong in Pillar 1's narrow EL scope.

## What this enables

- A clean separation of privileges: the **deploy role** has CREATE/GRANT privileges and applies DDL once per deploy; the **runtime role** is DML-only and runs the EL script. No elevated privileges in the daily-run path.
- DDL is reviewable in PRs at deploy time — humans see exactly what objects will be created or altered before approving.
- Idempotent application means CI re-runs and operator-driven retries are always safe.
- The pattern extends naturally to Pillar 2: dbt's CREATE OR REPLACE for models is fundamentally different from this idempotent additive style, but the *file-per-artifact + manifest-listed* structure carries over.
