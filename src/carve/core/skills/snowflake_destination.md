# Skill: snowflake_destination

Snowflake-specific destination patterns — `executemany` quirks,
`write_pandas`, `COPY INTO`, `MERGE` upsert, and the full DDL
emission contract for the per-EL companion file.

## `executemany` quirks

- Use `paramstyle="qmark"` on the connection. The `?` placeholder
  binds positionally, batches cleanly, and avoids the named-param
  edge cases.
- Batch size 1k–10k rows is the sweet spot. Larger batches increase
  memory pressure without speeding up the round-trip.
- `executemany` rejects Python `dict` and `list` bindings — convert
  to JSON strings via `json.dumps` before binding (see the
  `data_engineering` skill's "type coercion" section).
- The connection's `database` and `schema` set the default namespace,
  but explicit fully-qualified names (`<database>.<schema>.<table>`)
  in the SQL are clearer in audit logs and in the runner's stderr
  on failure.

```python
insert_sql = (
    "INSERT INTO ANALYTICS_DEV.RAW.IOWA_LIQUOR_SALES "
    "(invoice_line_no, date, store_number, sale_dollars) "
    "VALUES (?, ?, ?, ?)"
)
cursor.executemany(insert_sql, batch_of_tuples)
```

## `write_pandas`

Use when the load is medium-sized (10k–500k rows) and a DataFrame
is already in scope. `write_pandas` infers the schema from the
DataFrame, so types must already match the destination.

```python
from snowflake.connector.pandas_tools import write_pandas

success, n_chunks, n_rows, _ = write_pandas(
    conn=conn,
    df=df,
    table_name="IOWA_LIQUOR_SALES",
    database="ANALYTICS_DEV",
    schema="RAW",
    auto_create_table=False,  # rely on the DDL companion file instead
)
```

Pitfalls:

- `auto_create_table=True` creates a table with inferred types that
  rarely match what reviewers see in the DDL file. Keep it `False`
  and let `carve el deploy`'s DDL-apply phase create the table.
- Date columns must be `datetime64[ns]` (not Python `date`); pandas
  silently coerces nullable types to object and then write_pandas
  binds them as strings.

Adds `pandas` to `requirements.txt`. Do not add `pandas` for non-
write_pandas paths.

## `COPY INTO` from internal stage

For very large loads (>1M rows) where row-level binding is too slow.
PUT the file → `COPY INTO …` → cleanup. Used rarely in Pillar 1; the
DDL file emits an optional `CREATE STAGE IF NOT EXISTS` and `CREATE
FILE FORMAT IF NOT EXISTS` when this path is in use.

```sql
PUT file:///tmp/extract.csv.gz @~/carve_load/iowa_liquor_sales/;

COPY INTO ANALYTICS_DEV.RAW.IOWA_LIQUOR_SALES
FROM @~/carve_load/iowa_liquor_sales/
FILE_FORMAT = (TYPE = CSV FIELD_OPTIONALLY_ENCLOSED_BY = '"' SKIP_HEADER = 1)
ON_ERROR = ABORT_STATEMENT;

REMOVE @~/carve_load/iowa_liquor_sales/;
```

## `MERGE` upsert pattern

Canonical form for `merge_upsert`:

```sql
MERGE INTO ANALYTICS_DEV.RAW.IOWA_LIQUOR_SALES AS tgt
USING (
    SELECT
        column1 AS invoice_line_no,
        column2 AS date,
        column3 AS store_number,
        column4 AS sale_dollars
    FROM VALUES (?, ?, ?, ?)
) AS src
ON tgt.invoice_line_no = src.invoice_line_no
WHEN MATCHED THEN UPDATE SET
    date = src.date,
    store_number = src.store_number,
    sale_dollars = src.sale_dollars
WHEN NOT MATCHED THEN INSERT
    (invoice_line_no, date, store_number, sale_dollars)
    VALUES (src.invoice_line_no, src.date, src.store_number, src.sale_dollars);
```

Explicit column lists on both sides — relying on positional match is
brittle when the table evolves.

## Role / warehouse propagation

The connection's role and warehouse must reach the `connect()` call
explicitly. Do not let the connector default the role to the user's
default — the deploy role and runtime role are deliberately
separated, and the daily-run path must use the runtime role from
`<TARGET>_SNOWFLAKE_ROLE`.

## Snowflake-specific types

- `VARIANT` for JSON-ish nested data. Bind a JSON string via
  `json.dumps`; Snowflake parses it server-side.
- `NUMBER(p, s)` for numerics — match precision/scale to the source's
  reasonable maximum. `NUMBER(38, 0)` is the default; trim it down
  when the design's column declaration is more specific.
- `TIMESTAMP_NTZ` for "wall-clock" timestamps without timezone
  ambiguity. `TIMESTAMP_TZ` only when the source genuinely carries a
  timezone.
- `DATE` (no time part) for daily watermarks; `TIMESTAMP_NTZ` for
  fine-grained event time.

## DDL emission patterns (Pillar 1)

This is the contract for the per-EL companion file at
`targets/<active_target>/snowflake/<artifact>.sql`. The extract-load
agent emits this file on every build; `carve el deploy`'s DDL-apply
phase executes it.

### File location and naming

```
targets/<active_target>/snowflake/<artifact_name>.sql
```

One DDL file per artifact. The artifact name matches the EL pipeline
name (snake_case in the file path; the table name inside the SQL is
upper-snake-case by Snowflake convention).

### Required contents in order

1. Header comment block — three lines:
   - `-- Auto-generated by carve build for pipeline <name> (target: <target>).`
   - `-- Applied by `carve el deploy <name> --from <X> --to <target>` during its DDL-apply phase.`
   - `-- All statements are idempotent; re-running is safe.`
2. `-- === Schema ===` divider.
3. `CREATE SCHEMA IF NOT EXISTS <database>.<schema>;`.
4. `-- === Table ===` divider.
5. `CREATE TABLE IF NOT EXISTS <fqn> ( … );` with the design's
   columns. Inline `PRIMARY KEY (<col>)` if the design has one.
6. `-- === Grants ===` divider.
7. `GRANT SELECT, INSERT, UPDATE, DELETE ON TABLE <fqn> TO ROLE
   <runtime_role>;`.
8. (Optional) `-- === Stage ===` and `-- === File Format ===`
   sections when the script uses `COPY INTO` from an internal stage.

### Sample

```sql
-- Auto-generated by carve build for pipeline iowa_liquor_sales (target: dev).
-- Applied by `carve el deploy iowa_liquor_sales --from <X> --to dev` during its DDL-apply phase.
-- All statements are idempotent; re-running is safe.

-- === Schema ===
CREATE SCHEMA IF NOT EXISTS ANALYTICS_DEV.RAW;

-- === Table ===
CREATE TABLE IF NOT EXISTS ANALYTICS_DEV.RAW.IOWA_LIQUOR_SALES (
    INVOICE_LINE_NO  VARCHAR(50)   NOT NULL,
    DATE             DATE,
    STORE_NUMBER     VARCHAR(20),
    ITEM_DESCRIPTION VARCHAR(255),
    BOTTLES_SOLD     NUMBER(10,0),
    SALE_DOLLARS     NUMBER(12,2),
    LOCATION         VARIANT,
    _LOADED_AT       TIMESTAMP_NTZ DEFAULT CURRENT_TIMESTAMP(),
    PRIMARY KEY (INVOICE_LINE_NO)
);

-- === Grants ===
GRANT SELECT, INSERT, UPDATE, DELETE ON TABLE ANALYTICS_DEV.RAW.IOWA_LIQUOR_SALES
    TO ROLE TRANSFORMER_DEV;
```

### Idempotency contract

Always allowed:

- `CREATE … IF NOT EXISTS` (schema, table, stage, file format)
- `GRANT …` (Snowflake grants are idempotent)
- `ALTER TABLE … ADD COLUMN IF NOT EXISTS`

Allowed only with explicit plan-time approval (the design's
`tradeoffs` array names the destructive change):

- `DROP TABLE IF EXISTS` / `DROP COLUMN IF EXISTS`
- `DROP SCHEMA IF EXISTS … RESTRICT` (cascade is forbidden)
- Paired `DROP IF EXISTS; CREATE IF NOT EXISTS;` for replacement
  intent

Always forbidden:

- `CREATE OR REPLACE …` — overwrites silently. Use the paired
  `DROP IF EXISTS; CREATE IF NOT EXISTS;` form instead, which is
  more legible at PR-review time.
- Bare `RENAME` (`ALTER TABLE … RENAME`) — Snowflake has no
  idempotent form. If a rename is requested, surface the
  conflict via `submit_step(error=True)`.
- `ALTER COLUMN … SET DATA TYPE …` — lossy when data is already
  present.
- DML embedded in DDL (e.g. `INSERT INTO migrations_log …`).

### Modify-existing-pipeline path

When the agent is invoked with `action="modify_extractor"` and the
design adds a new column, emit `ALTER TABLE <fqn> ADD COLUMN IF NOT
EXISTS <name> <type>;` rather than `CREATE OR REPLACE TABLE`. The
existing `CREATE TABLE IF NOT EXISTS` block remains in the file (the
no-op on re-apply is the point of "idempotent").

### Destructive intent surfaces in `tradeoffs`

If the design's `tradeoffs` array contains a string naming a
destructive change (for example "Will drop existing column foo …"),
the agent emits the corresponding `DROP COLUMN IF EXISTS` or `DROP
TABLE IF EXISTS` statement. If the destructive change is not named
in `tradeoffs`, the agent must not emit it — call
`submit_step(error=True)` instead and let the user re-plan.

### File contents are concrete, not parameterized

The DDL file references concrete database and role values (resolved
from the active target's `[snowflake.<target>]` section). Do not
emit `${ENV_VAR}` placeholders — the file is committed as a snapshot
of "what this build expects in this target." If the user changes
`connections.toml` after the build, they re-build to regenerate the
DDL.
